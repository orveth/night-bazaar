# Night Bazaar

> **Status & build note.** A working demo on the pops/cashu accept-layer stack —
> pay pops at a gate, win ecash. It is published to read and share; it is not yet
> a standalone buildable checkout, because the server depends on the **private**
> `MakePrisms/pops` repo (the `pops-core-verify` middleware + charge-01 codec), so
> `cargo build` needs access to that repo. The `../plans/*.md` design contracts
> referenced below live in a separate workspace and are not included here.

One Rust axum app serving the game websocket (server-authoritative
positions + jump, per-session entitlements, chat relay), the pops-gated
endpoints (`POST /spawn`, `POST /enter/:court`, and the Phase-1b paid plays
`POST /play/gacha` + `POST /play/bell` behind pops-core-verify middleware @
MakePrisms/pops `28e3314`), and the built Three.js client.

Phase 0 = the spine (contract `../plans/night-bazaar-phase0-spike.md`, rev 2).
Phase 1a = THE LOOK (contract `../plans/night-bazaar-phase1a-look.md`):
night-market art pass (lantern strings, emissive-MRT bloom, fog, 12 stalls,
gate signage, steam, synthesized ambience), procedural alien avatars, ghost
wisps, proximity chat bubbles, jump, and 3 hidden street finds (rooftop /
behind-a-stall / alley-corner chests).
Phase 1b = PLAYABLE BOOTHS + the revenue sink (contract
`../plans/night-bazaar-phase1b-booths.md`): three booths — the riddle lantern
(jade court, FREE; interact opens a modal, a correct answer wins a prize and
rotates the riddle), the gacha shrine (crimson court, paid per pull,
deterministic every-Nth-wins via a server counter), the timing bell (street,
paid per play, server-clock-judged pendulum) — plus the durable revenue sink:
every redeemed gate/play proof is appended to `vault/revenue.jsonl` (a WALLET)
BEFORE the grant/result returns. The `chest.stall` line-of-sight quirk (it was
claimable through the trinket stall wall) is fixed by a segment-vs-occluder
check. Design: `../plans/night-bazaar-design-v1.md`.

Client controls: WASD move · [B] buy a body · [E] interact/pay/play · [Space]
jump (the rooftop find needs it) · Enter/T chat (bodies only) · [M] mute. At a
booth [E] reads the riddle / pulls the gacha / starts the bell; during a bell
play [E] RINGS it; in the riddle modal type your answer + Enter.
Dev query params: `?webgl=1` force the WebGL2 backend (default = try WebGPU,
auto-fallback), `?nobloom=1` skip the post chain, `?crowd=N` client-side
fake wanderers for screenshots/FPS (never networked).

## Layout

- `server/` - cargo crate (`night-bazaar-server` bin + lib + `gen-vectors` bin)
- `client/` - bun + Three.js (WebGL for the spike); fresh charge-01 payer codec
  in `client/src/charge01.ts`, written per the pops repo's
  `skills/payment-credential.md`
- `protocol/` - the seam: `protocol.ts` (TS source of truth, mirrored by
  `server/src/protocol.rs`), shared fixtures, and golden charge-01 vectors
  generated by pops' own encoders (both test suites consume them)
- `vault/tokens.json` - prize stock, keyed by chest id AND booth id:
  `{"chest.jade": ["cashuB…"], "chest.rooftop": [], "booth.riddle": [],
  "booth.gacha": [], "booth.bell": []}` (BEARER cash, keep small, never
  commit). The Phase-0 format (a bare JSON array) still READS as
  `chest.jade`'s stock and is upgraded in place on the first claim. A chest
  with no stock renders as already-looted from boot; a booth with no stock
  renders "out of prizes" on a win (the win/rotate still happens, the player
  just gets no token). The three `booth.*` keys ship EMPTY — the keeper
  stocks them (the file is read at claim time); restock + restart to bring
  any chest/booth to life.
- `vault/revenue.jsonl` - the durable revenue sink (a WALLET, never commit):
  one JSONL line per redeemed gate/play, `{ts, gate, session, amount, unit,
  proofs}`, where `proofs.fresh_proofs` is the spendable `cashuB…` the
  operator now holds. Written + fsynced BEFORE the grant/play-result returns
  (a lost proof is lost money). Back it up; restrict access.
- `.local/` - test funds and run artifacts (never commit)

## Run

Server (from `server/`; cargo comes from the pops devshell; the git dep on
the private pops repo needs the CLI fetcher):

```sh
cd server
CARGO_NET_GIT_FETCH_WITH_CLI=true nix develop /srv/forge/projects/pops -c cargo run --bin night-bazaar-server   # two bins in this package; bare `cargo run` errors. Prebuilt: ./target/debug/night-bazaar-server
```

Knobs (env): `BAZAAR_BIND` (default `100.96.251.111:8410`), `BAZAAR_MINT_URL`
(default `http://127.0.0.1:28338`), `BAZAAR_MODE` (`live` default | `mock` =
free gates + free plays, dev/smoke only), `BAZAAR_PRICE_SPAWN/_JADE/_CRIMSON`
(10/50/200), `BAZAAR_PRICE_GACHA/_BELL` (5/3, paid plays), `BAZAAR_GACHA_N`
(8 — gacha wins every Nth pull, deterministic), `BAZAAR_VAULT`,
`BAZAAR_REVENUE_SINK` (default `../vault/revenue.jsonl` — the durable redeemed-
proofs sink, a WALLET; in live mode the server refuses to boot if it cannot
open it), `BAZAAR_STATIC`, `BAZAAR_BINDING_KEY` (hex; unset = per-boot key),
`BAZAAR_CHALLENGE_TTL_SECS`, `BAZAAR_MINT_TIMEOUT_SECS`,
`BAZAAR_UNIT_REFRESH_SECS` (300 — how often the valid pop-unit set is
re-probed from the mint, for rotation).

**Multi-unit accept (rotation-safe).** `pop_<ts>` units rotate and OVERLAP, so
the server tracks the whole SET of currently-valid units, not one. The valid set
= every `pop_<ts>` unit from `/v1/keysets` whose `final_expiry` is in the FUTURE
(the `active` flag is IGNORED — it lies: a retired keyset reads `active:true`,
and an undated keyset is served for dead units; both traps are dodged). The
NEWEST (latest `final_expiry`) is advertised in `/api/config` as `unit` (what a
fresh player mints into) plus `acceptedUnits` (the full set). A gate request
DECLARES the unit it holds (`?unit=pop_<ts>` query, or `x-bazaar-unit` header);
the server issues the challenge IN THAT UNIT if it is valid, so an
older-but-unexpired unit is honored instead of cutting the player off. A held
unit that has expired → a clear `unit-retired` 409 ("mint the current unit").
This is a dispatch layer over the UNMODIFIED pops verifier (it still does its
strict per-challenge `token_unit == requirement.unit` check — we only choose
WHICH unit's challenge to issue; `server/src/multiunit.rs`). The set is
refreshed every `BAZAAR_UNIT_REFRESH_SECS` (default 300) by a background
re-probe. Never hardcoded.

## Deploy (Fly.io)

A clickable public URL, no client install. The image is THIN (a prebuilt binary
+ the built client + a bundled glibc — the private pops dep is never built in
Docker, so no SSH key is baked). Wallet files (`tokens.json`, `revenue.jsonl` —
real money) live on a persistent Fly **volume** at `/data`, never in the image;
`BAZAAR_BINDING_KEY` is a Fly **secret** so deploys keep outstanding challenges
valid. Build the artifacts, then deploy:

```sh
bash server/build-image.sh                 # release build + bun build + staged runtime tree
docker build -t night-bazaar:latest .      # thin runtime image (distroless/static, ~40 MB)
```

Full ordered `fly` command sequence (app/volume/secret/deploy/stock) is in
[`docs/fly-deploy.md`](docs/fly-deploy.md). The deploy mint is
`poptest.agi.cash` (mainnet). To smoke the image locally against the shared
Mutinynet rig, run it with `--network host` and
`BAZAAR_MINT_URL=http://100.96.251.111:28338` (see the guide's last section).

Client (from `client/`):

```sh
bun install
bun run build        # bundles src/main.ts + index.html into dist/ (served by the server)
bun test             # codec vs golden vectors, wallet, protocol fixtures
bun x tsc -p tsconfig.json --noEmit
```

Server tests (from `server/`):

```sh
CARGO_NET_GIT_FETCH_WITH_CLI=true nix develop /srv/forge/projects/pops -c cargo test
```

Golden vectors regenerate with
`cargo run --bin gen-vectors > ../protocol/vectors/charge01-vectors.json`
(a cargo test fails if the committed file drifts from pops' encoders).

## Money path (proven 2026-06-10 vs the shared Mutinynet rig on :28338)

- pop-pay CLI first pass: `pop pay "http://100.96.251.111:8410/spawn?session=<ws session>" --method POST --token-file <bankroll> --max-amount 15 --wallet-dir <rig wallet>`
  (the `?session=` query is the CLI affordance; browsers use the
  `x-bazaar-session` header)
- Full matrix: `bun client/scripts/live-money.ts <serverBase> <bankrollFile> <outDir> <vaultFile>`
  (spawn + replay-rejected + jade + wrong-amount-rejected + chest claim)
- Paid booths (Phase 1b): `bun client/scripts/keeper-booth-paytest.ts <serverBase> <bankrollFile>`
  (spawn -> enter crimson -> walk to the gacha shrine -> ONE paid pull; then
  the street bell -> ONE paid play -> press; banks change on every exit path).
  Proven 2026-06-10: gacha pull + bell play+press both returned results, four
  `vault/revenue.jsonl` lines appended (spawn/crimson/gacha/bell).
- Browser payer: import a cashuB in the HUD, `[B]` spawn, `[E]` at a door or
  booth, type the riddle answer, time the bell.

Chest/booth restock: carve a small token from the live bankroll into
`vault/tokens.json` under the chest's or booth's id (see Layout) and restart
the server (the claimed flag derives from stock at boot; the vault file is
read at claim time).
